# Task System App

A real-world example application for `nestjs-yalc`.

This project is intended to serve **three roles at the same time**:

1. **reference backend** showing how `nestjs-yalc` should be used in a realistic modular application;
2. **integration test bed** for unit, integration, and e2e coverage across the framework;
3. **standalone task management backend** that is useful on its own and can later be consumed by external orchestrators or clients.

## Primary goals

- Exercise `crud-gen` against a non-trivial domain.
- Exercise `api-strategy` with module-to-module communication.
- Exercise `event-manager` / `YalcEventService` in real workflows.
- Provide a complete demo backend instead of a toy example.
- Keep the domain model provider-agnostic so external integrations (Google Tasks, Calendar, ClickUp, etc.) remain adapters, not the core model.

## Architectural role

This application is the **domain/backend core**.

It must be:

- standalone;
- runnable;
- end-to-end testable;
- usable without any external assistant layer.

External consumers may orchestrate or extend it later, but they must not be required for the app to be useful.

## Domain boundaries

The initial version should be split into bounded modules rather than a single flat task module.

Suggested modules:

- `tasks`
  - core actionable items
  - status, due dates, completion, hierarchy
- `projects`
  - project containers and project-level lifecycle
- `areas`
  - long-lived responsibility buckets
- `events`
  - real calendar-like events, distinct from tasks
- `sync`
  - external references and synchronization state
- `inbox` (optional early, likely useful)
  - raw captured items before triage

## Core modeling rules

These rules should remain stable:

- **Task** and **Event** are different concepts.
- Internal IDs are generated by the backend and never derived from external providers.
- External systems are represented through `externalRefs`, not by leaking provider IDs into the core model.
- Domain services must not depend directly on provider-specific semantics.
- Cross-module communication should go through strategies/tokens, not direct provider coupling.

## Recommended `nestjs-yalc` usage

### CRUD/API surfaces

Use `crud-gen` wherever the domain fits standard CRUD patterns:

- `CrudGenDependencyFactory`
- `crudRestControllerFactory`
- generated GraphQL providers/resolvers when GraphQL exposure is useful

### Module communication

Use `api-strategy` for module/domain boundaries:

- local runtime: `NestLocalCallStrategyProvider`
- remote/service mode: `NestHttpCallStrategyProvider`
- async flows: event strategies

### Logging and errors

Use `YalcEventService` consistently for:

- structured logs
- domain events
- HTTP-aware errors
- error forwarding/normalization

## Testing role inside `nestjs-yalc`

This app should become the strongest full-stack example in the repository.

Suggested testing pyramid:

- **unit tests**
  - service-level domain behavior
  - mapping and validation logic
- **integration tests**
  - module wiring
  - repository/service/controller interplay
  - strategy-based communication between modules
- **e2e tests**
  - realistic workflows across the application
  - CRUD flows
  - event/log/error flows
  - sync-oriented flows

## CI/CD value

Once implemented, this app should be included in repository checks as a real validation target for:

- build correctness
- framework compatibility
- generated CRUD surfaces
- event/error behavior
- strategy-based transport behavior

This makes the project useful both as a demo and as a regression safety net for `nestjs-yalc` itself.

## Consumer boundary

Assistant layers, automations, third-party UIs, or external sync workers should stay outside this app.

This backend is responsible for:

- canonical domain model
- validation
- persistence
- provider-agnostic business logic
- stable API for task/event/project management

## Incremental build plan

### Phase 1

Create the real example app skeleton with these first modules:

- `tasks`
- `projects`
- `events`
- `sync`

### Phase 2

Add tests proving:

- CRUD generation works in a realistic domain
- task/event separation is respected
- strategies can isolate module boundaries
- `YalcEventService` is used coherently

### Phase 3

Add provider-facing integration surfaces (still provider-agnostic in the core), e.g.:

- external refs
- sync state
- import/export orchestration contracts

## Non-goals for the first implementation

Do **not** start with:

- UI/dashboard work
- provider-specific coupling in the core model
- premature workflow engines
- speculative microservice splitting
- heavy infrastructure not required by the example

## Outcome

If implemented well, this app becomes:

- the best real-world `nestjs-yalc` example in the repo;
- a durable CI/e2e validation target;
- a reusable standalone backend for task and project management.

## OmniKernel adoption slice

The task app now includes an explicit OmniKernel adapter layer for the first
adoption slice.

Current scope:

- REST writes for `projects`, `tasks`, and `external-refs` persist into
  OmniKernel entities through a dedicated adapter layer
- project containers map to `OmniCollectionEntity`
- task items map to `OmniRecordEntity` with `kind = 'task'`
- external refs map to `OmniExternalRefEntity`
- project membership uses canonical `contains` relations
- optional task cross-links use `references` and `related_to`
- the same REST writes also mirror to the legacy task tables for temporary
  compatibility with the unchanged event and GraphQL slices during migration

Current non-goal:

- the generated GraphQL CRUD layer still uses the legacy task-system module
  entities for now; the OmniKernel adoption is intentionally incremental and
  focused on a reviewable adapter-backed REST path first

The adapter lives under
`apps/task-system-app/src/omni-task-app/` and keeps the mapping rules explicit
instead of scattering Omni-specific logic through controllers.
